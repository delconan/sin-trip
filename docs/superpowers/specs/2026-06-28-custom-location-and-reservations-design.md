# 自定义地点与预约状态设计

## 目标

修复自定义活动“地址文字正确但坐标回退酒店”的问题，并让用户可以直接修正既有自定义卡、明确删除自定义卡。同时为景点和美食卡增加可切换、可同步的预约状态。

## 自定义地点

### 新建

- 地址输入满 3 个字符后继续使用 OneMap 搜索，保留 250ms 防抖。
- 用户点击候选地址时保存地址、纬度和经度。
- 提交时若已有有效经纬度，直接使用。
- 若只有地址文字，提交时再次查询完整地址：唯一结果自动采用；多个结果展示候选并提示用户选择；零结果或请求失败则阻止保存并显示错误。
- 删除现有的酒店坐标兜底。未确认坐标的自定义卡不能保存。
- 用户重新编辑地址文字后，旧的候选坐标立即失效，避免地址与坐标不一致。

### 编辑

- 自定义卡详情页的“地点”改为可编辑区域，复用与新建卡相同的 OneMap 地址选择逻辑。
- 保存地点后以卡片级动作更新 `address`、`latitude`、`longitude`，所有已排入行程的实例自动使用新坐标。
- 路线查询键已经包含经纬度，坐标更新后相关交通段会重新查询 OneMap。
- 非自定义预置卡保持只读。

## 删除自定义卡

- 保留现有 `delete-card` 状态动作；它会同时删除卡片及该卡的全部行程实例。
- 将删除入口放在自定义卡详情页的明显位置。
- 首次点击进入内联确认状态，显示“确认删除”和“取消”；只有再次确认才执行删除。
- 预置卡不显示删除入口。

## 预约状态

- `ActivityCard` 增加可选的 `reservationStatus: "required" | "booked"`。
- 所有景点和美食卡默认迁移为 `required`；购物、休息、交通卡不显示预约状态。
- 候选库卡片、每日行程卡片和详情页使用同一个卡片级状态。
- 状态按钮初始显示“需预约”，点击后显示“✓ 已预约”，再次点击恢复“需预约”。
- 切换通过 reducer 更新卡片并增加行程 revision，因此本地持久化及 Supabase 共享同步沿用现有状态通道。
- 自定义景点或美食卡创建后也默认显示“需预约”。

## 组件边界

- 提取可复用的地点选择组件，负责地址草稿、OneMap 候选、坐标有效性和错误展示；新建弹窗与详情编辑共同使用。
- `trip-state.ts` 增加卡片地点更新和预约状态切换动作，保持状态变更集中、可测试。
- 卡片 UI 只负责触发动作和显示状态，不自行复制预约状态。

## 错误处理

- OneMap 无结果、网络失败或存在多个未选择结果时不保存，并保留用户输入。
- 地点保存期间禁用确认按钮，避免重复请求。
- 删除操作必须二次确认。
- 预约状态切换为本地即时操作，不依赖 OneMap。

## 测试与验收

- reducer：更新地点会保留卡片身份并增加 revision；预约切换可往返；删除卡会移除全部实例。
- 组件：唯一地址自动解析、多结果要求选择、零结果阻止保存、详情可修改地点、删除二次确认。
- 组件：景点和美食显示预约按钮，其他类别不显示；候选卡、行程卡与详情状态同步。
- 端到端：修正 No Signboard Seafood 为 `414 GEYLANG ROAD SINGAPORE 389392` 后，交通链接目的地使用 `1.312888405679514,103.8826252657034`，不再指向 City Hall。
- 端到端：预约状态切换后刷新页面仍显示“已预约”。
- 完整运行单元测试、类型检查、Lint、生产构建和桌面/移动端 Playwright。
