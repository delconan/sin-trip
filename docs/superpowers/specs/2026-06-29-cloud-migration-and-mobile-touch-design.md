# 云端安全迁移与手机触屏行程设计

## 目标

解决已部署应用中的两个问题：第一，电脑已有本地修改时，Supabase 初始化不能用默认底稿覆盖本地行程，并且分享操作必须生成带私密令牌的完整链接；第二，手机端不能依赖桌面四列看板的横向指针拖拽，而要提供适合触摸、滚动和跨日移动的操作。

本次改造必须保证当前电脑浏览器中仍然存在的本地行程，在新版本首次加载时可以安全迁移到云端。应用继续使用匿名 Supabase 会话和 URL fragment 中的 128-bit 私密令牌，不引入账号密码。

## 同步启动状态

同步启动拆分为明确的状态机，不再让本地恢复和云端连接两个副作用竞争：

1. `loading-local`：先读取并校验 `localStorage`，此时不访问行程创建接口。
2. `needs-cloud-action`：网址没有有效令牌时，保留当前本地状态并等待用户选择“保存当前行程到云端”或“创建新的初始行程”。
3. `joining`：网址带有效令牌时，建立匿名会话并加入该云端行程；云端状态是权威来源。
4. `creating`：用户明确确认后，把当前本地快照或默认底稿作为新行程初始状态上传。
5. `synced` / `saving` / `error`：加入成功后沿用版本号与 Realtime 同步；错误状态保留本地数据，并显示可理解的具体原因和重试入口。

`PlannerApp` 负责本地恢复完成标记，并把稳定的本地快照交给 `useTripSync`。`useTripSync` 在标记完成前不得创建或加入行程，从根源上消除默认状态覆盖本地状态的竞态。

## 本地行程迁移

- 网址没有令牌时，页面顶部显示持久提示，而不是静默调用 `/api/trips/create`。
- 如果浏览器已有保存状态，主按钮显示“保存当前行程到云端”；没有保存状态时显示“创建共享行程”。
- 上传前把当前 JSON 快照保存到独立备份键 `singapore-family-trip-v1-backup`，包含备份时间。只有云端创建和随后读取均成功，界面才宣布迁移完成；备份保留，供意外恢复使用。
- `POST /api/trips/create` 接收可选的 `state`。服务端验证 revision、日期标题、卡片和行程项目的结构、数量及引用关系，拒绝畸形或过大的请求。
- 创建事务应完整写入 `trips`、`trip_members`、`activity_cards` 和 `scheduled_items`。任一步失败时删除本次不完整行程，避免残留半成品。
- 创建成功后，客户端用 `history.replaceState` 把地址更新为 `/trip#<token>`，并以服务端返回状态 hydrate。分享按钮只在存在 `shareUrl` 时复制私密编辑链接。
- 新设备只打开首页时，页面提供私密链接/令牌输入和“创建新行程”两个明确入口；它不能猜测用户属于哪一份私密行程，也不能自动显示另一台设备的数据。

## 分享和错误反馈

- “分享”在尚未建立云端行程时不再复制首页地址，而是把用户引导到“保存当前行程到云端”。
- 已同步后复制的地址固定为 `https://域名/trip#<32位令牌>`。
- 同步徽标之外增加错误说明，例如“匿名登录未启用”“服务端凭据缺失”“数据库表尚未建立”“分享链接无效”或“版本冲突，请重试”。控制台仍记录原始错误，但界面不暴露密钥或内部堆栈。
- 粘贴入口接受完整分享 URL 或单独的 32 位令牌；验证后在当前标签打开 `/trip#令牌`。

## 手机日期视图

- 桌面端继续显示四日看板。
- 手机“每日行程”标签页顶部增加四个日期标签，一次只渲染选中日期的完整宽度行程列，减少横向滚动和误放。
- 日期标签显示当天活动数量；移动活动到另一日后自动切换到目标日期并聚焦移动后的卡片。
- 卡片库的“加入某天”快捷操作保留。

## 触摸拖拽和替代操作

- 桌面端继续使用带移动阈值的 `PointerSensor`。
- 手机端增加 `TouchSensor`，专用拖拽把手长按约 250ms、允许约 8px 手指抖动后才激活；在卡片正文和页面空白处滑动仍然滚动页面。
- 手机一次只显示一天，因此长按拖拽负责当天内部排序。拖拽期间继续使用现有 `DragOverlay` 和落点预览，松手时只提交一次状态变更。
- 每张已安排卡片增加移动菜单。选择“移动到某天”后，在同一面板内选择目标日期和 15 分钟粒度开始时间；确认后一次性提交并按开始时间重排。
- 提供键盘和触摸可用的“上移/下移”按钮。按钮与相邻活动交换排序；若开始时间不同，确认面板会显示调整后的目标开始时间，用户确认后才提交，避免无提示改动时间。
- 时间编辑继续使用草稿：输入过程中卡片位置不变，只有 Enter 或确认按钮会保存并重排。

## 数据一致性与失败处理

- 分享令牌继续只以哈希形式存入数据库，fragment 不发送给普通页面请求。
- 加入带令牌的行程时不合并当前设备的本地默认数据，避免污染云端；加入成功后云端覆盖当前设备视图。
- 创建当前本地行程时只上传一次稳定快照。上传过程中发生的后续编辑要在创建成功后以正常 revision 更新继续保存。
- 网络失败、匿名登录失败或创建失败均不得清除本地状态或备份；用户可点击重试。
- Realtime 断开时保留本地编辑并显示状态，恢复后沿用原子 revision 接口处理冲突，不做静默覆盖。

## 测试与验收

- 单元测试：本地状态识别、分享 URL/令牌解析、创建负载校验、备份数据、启动状态转换、非法创建负载拒绝。
- Reducer/组件测试：本地恢复完成前不创建；无令牌显示迁移入口；分享按钮不会复制首页；日期标签切换；移动面板跨日和定时；时间草稿不即时重排。
- API 集成测试：当前本地快照可完整创建为云端行程；写入失败能清理半成品；成员权限与版本号正确。
- Playwright 双浏览器测试：设备 A 从已有本地修改创建共享行程，地址生成令牌；设备 B 使用完整链接加入并看到完全相同状态；任一设备修改后另一设备收到更新。
- Pixel 7 触摸测试：普通滑动可滚动，短触不拖动，长按把手可在当天排序，移动面板可跨日，桌面鼠标拖拽不回归。
- 发布前运行类型检查、Vitest、生产构建和桌面/手机 Playwright。发布完成后，原电脑刷新页面，选择“保存当前行程到云端”，确认地址带令牌并复制到手机。

## 不在本次范围

- 不增加邮箱、手机号或第三方账号登录。
- 不让仅打开首页的陌生设备自动发现现有私密行程。
- 不改变现有景点、美食、路线、价格或预约数据模型。
- 不自动部署到 Vercel，也不替用户操作生产 Supabase 数据；交付可部署代码与明确迁移步骤。
